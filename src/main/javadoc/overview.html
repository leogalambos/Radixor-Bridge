Radixor-Bridge

Radixor-Bridge is a dedicated Apache Lucene 10 adapter for Radixor. It enables the use of Radixor stemming models inside Lucene analysis pipelines while keeping Lucene dependencies out of the Radixor core project.

The module is intentionally narrow in scope. Radixor remains responsible for stemming data, patch-command encoding and application, dictionary parsing, compiled trie persistence, and runtime stemming semantics. Radixor-Bridge is responsible only for Lucene-facing integration: token-stream processing, analyzer wiring, factory-based configuration, and SPI-oriented loading.

Purpose

This project is intended for search systems that want to use Radixor stemming in Lucene-based analysis chains without embedding Lucene concerns into the core stemming toolkit. It is especially relevant in environments where one would otherwise consider:

  • Stempel-style stemming,
  • Porter-family stemmers for multiple languages, or
  • a compact dictionary-trained stemmer that remains operationally simple.

What the adapter provides

  • A Lucene TokenFilter that applies Radixor stemming to token terms.
  • A Lucene TokenFilterFactory suitable for SPI and CustomAnalyzer usage.
  • A Lucene Analyzer convenience wrapper for direct programmatic use.
  • Configurable model activation paths for bundled languages, text resources, binary resources, and binary files.
  • Deterministic runtime behavior aligned with the Radixor compiled trie model.

Runtime behavior

The runtime operation is intentionally small and direct:

  1. Read the current token from Lucene's CharTermAttribute.
  2. Look up the token in a Radixor patch-command trie.
  3. Apply the returned patch command to the original token.
  4. Replace the term only when the transformation is valid and meaningful.

Tokens marked with Lucene's keyword attribute are preserved unchanged. Unknown terms, malformed patch commands, empty results, and identity transformations also preserve the original token rather than introducing surprising behavior into the analysis chain.

Configuration model

Radixor-Bridge is designed for ergonomic multi-language use. A stemmer can be activated in several ways, with the default configuration using the bundled US_UK_PROFI model.

  • Default bundled language when no explicit source is configured.
  • Explicit bundled language via a language parameter.
  • Classpath text dictionary resource for dictionary-driven loading.
  • Classpath binary resource for compiled binary model loading.
  • Filesystem binary path for operational deployment of compiled artifacts.
  • Custom provider override for advanced or application-specific loading strategies.
For ordinary use, applications should prefer the built-in configuration paths rather than custom providers. Custom providers are intended mainly for advanced integrations, controlled deployments, or specialized test scenarios.

Design principles

  • Separation of concerns: Radixor handles stemming; Radixor-Bridge handles Lucene integration.
  • Operational clarity: model loading is explicit and predictable.
  • Determinism: token transformation follows Radixor's compiled stemming semantics.
  • Compactness: the adapter remains small instead of becoming a second stemming engine.
  • Maintainability: Lucene-specific evolution is isolated from the core stemming implementation.

Typical usage

Programmatic analyzer usage

Analyzer analyzer = RadixorAnalyzer.forDefaultBundledLanguage();

Lucene CustomAnalyzer usage

Analyzer analyzer = CustomAnalyzer.builder()
    .withTokenizer("standard")
    .addTokenFilter("radixor")
    .build();

Explicit language selection

Analyzer analyzer = CustomAnalyzer.builder()
    .withTokenizer("standard")
    .addTokenFilter("radixor", "language", "DE_DE")
    .build();

Project scope

Radixor-Bridge is not a full morphological analysis framework. It is a focused integration layer for applying Radixor stemming models inside Lucene. Its value lies in a fast, compact, deterministic, and operationally practical stemming path for real search systems.
